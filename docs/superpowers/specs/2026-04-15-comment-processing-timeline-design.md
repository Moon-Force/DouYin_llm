# 评论处理时间线设计

## 目标

把事件流里“评论处理状态”的展示，从一排并列标签升级成“处理轨迹”。

这次改动要解决两个问题：

- 用户第一眼能看懂一条评论当前走到了哪一步
- 用户展开后能看到这条评论完整经历了什么，以及哪些步骤没有发生

本设计只改前端展示层，不改后端处理逻辑和 `processing_status` 数据结构。

## 适用范围

仅适用于事件流中的评论事件卡片。

包含：

- 评论卡片底部的状态区改为“混合式时间线”
- 默认显示横向概览时间线
- 展开后显示纵向详细轨迹
- 保留现有调试字段展示能力，但改成轨迹信息的一部分

不包含：

- 非评论事件的状态展示重做
- 后端新增状态字段
- 独立的评论详情页
- 历史事件补算轨迹

## 设计原则

- 主视图优先回答“发生了什么”，不要堆满失败提示
- 展开视图补全“没发生什么、为什么没发生”
- 中文文案短而清楚，适合快速扫读
- 延续现有事件卡片视觉语言，不做整页风格重构

## 方案比较

### 方案一：横向概览 + 纵向详情

评论卡片默认显示一条横向时间线，展示关键处理节点。点击“处理详情”后，在下方展开纵向轨迹，逐步展示状态说明和关联数据。

优点：

- 第一眼易扫读
- 展开后信息完整
- 最符合“处理轨迹”而不是“标签堆叠”的目标

缺点：

- 实现量略高于纯样式替换

### 方案二：纯纵向轨迹

默认直接展示完整纵向步骤流。

优点：

- 轨迹感最强
- 文案表达空间更大

缺点：

- 评论列表会显著变高
- 实时事件多时不利于快速扫读

### 方案三：流程化标签

保留现有一排状态项，只把视觉改成连接式流程条。

优点：

- 改动小
- 风险低

缺点：

- 本质上仍像标签
- 不足以体现“这条评论经历了什么”

### 推荐方案

采用方案一：横向概览 + 纵向详情。

这是信息密度、扫读效率和处理轨迹感之间最平衡的方案。

## 交互设计

### 默认态

评论卡片正文下方显示一条紧凑横向时间线，节点顺序固定为：

1. 已接收
2. 评论入库
3. 写入记忆
4. 参与召回
5. 生成建议

时间线承担“概览”职责：

- 已完成步骤高亮显示
- 未命中或未生成步骤使用中性样式
- 跳过步骤弱化显示
- 节点之间用连接线体现处理顺序

主时间线仍保留“处理详情”开关。

### 展开态

点击“处理详情”后，在评论卡片下方展开纵向轨迹。

纵向轨迹按时间顺序展示每一步，包含：

- 步骤名称
- 状态结果
- 一句短说明
- 可选关联数据

例如：

- 评论已入库：已写入 SQLite
- 观众记忆写入：未产生可保存记忆
- 观众记忆召回：命中 2 条记忆
- 回复建议生成：已生成 suggestion_id

### 未发生步骤的显示策略

按已确认的产品方向，采用折中策略：

- 主时间线以“发生了什么”为主
- 展开详情中明确写出“未发生 / 未命中 / 已跳过”

这样既不会让卡片默认状态过吵，也不会丢失排查价值。

## 状态映射

前端继续基于现有 `processing_status` 生成展示数据，不要求后端新增字段。

建议把状态映射收敛成统一的时间线节点结构，例如：

```js
{
  key: "memory_recall",
  title: "参与召回",
  state: "success" | "neutral" | "skipped",
  summary: "命中 2 条记忆",
  detail: "recalled_memory_ids: mem-1, mem-2"
}
```

各步骤映射建议如下：

- 已接收
  - `received=true` 时标记为 `success`
- 评论入库
  - `persisted=true` 为 `success`
  - 否则为 `neutral`
- 写入记忆
  - `memory_saved=true` 为 `success`
  - `memory_extraction_attempted=true && memory_saved=false` 为 `neutral`
  - `memory_extraction_attempted=false` 为 `skipped`
- 参与召回
  - `memory_recalled=true` 为 `success`
  - `memory_recall_attempted=true && memory_recalled=false` 为 `neutral`
  - `memory_recall_attempted=false` 为 `skipped`
- 生成建议
  - `suggestion_generated=true` 为 `success`
  - 否则为 `neutral`

## 文案设计

全部使用中文短文案。

主时间线节点文案建议：

- 已接收
- 已入库 / 未入库
- 已写记忆 / 未写记忆 / 已跳过
- 已召回 / 未召回 / 已跳过
- 已生成建议 / 未生成建议

展开轨迹说明建议：

- 系统已接收该评论并进入处理流程
- 评论已成功写入本地存储
- 已尝试提取观众记忆，但没有可保存内容
- 已参与语义召回，但没有命中观众记忆
- 已生成回复建议，可用于后续主播话术

如果存在关联 ID，则在对应步骤下显示：

- `saved_memory_ids`
- `recalled_memory_ids`
- `suggestion_id`

## 组件与职责

### `event-feed-processing-presenter.js`

从“返回一组标签”调整为“返回时间线节点和详情节点”。

建议新增两个面向展示层的方法：

- `getCommentProcessingTimeline(event)`
- `getCommentProcessingDetails(event)`

其中：

- `getCommentProcessingTimeline()` 返回横向概览节点
- `getCommentProcessingDetails()` 返回纵向轨迹条目

如果需要，可保留旧方法作为过渡，但最终目标是由时间线数据驱动模板，而不是由标签列表驱动。

### `EventFeed.vue`

负责：

- 渲染横向概览时间线
- 保留并复用现有展开态控制
- 渲染纵向详细轨迹
- 在无 `processing_status` 时不展示时间线区域

不负责：

- 直接写死复杂状态判断

状态判断应尽量留在 presenter 层，保持模板可读性。

## 视觉建议

### 横向概览

- 每个步骤用“圆点 + 标签”表示
- 步骤之间有细连接线
- 成功态使用较强强调色
- 未命中态使用中性色
- 跳过态使用更淡的虚线或弱化样式

### 纵向轨迹

- 左侧为竖向连线与节点
- 右侧为步骤标题、状态说明和关联数据
- 每个步骤之间保留足够呼吸感，避免像表格

### 可读性要求

- 移动端下允许横向概览换行，但仍应保持顺序清晰
- 不能让时间线压过评论正文
- 非评论事件卡片视觉不受影响

## 异常与边界

- 若事件缺少 `processing_status`，则不显示时间线
- 若步骤执行过但没有产出，显示为“未命中”或“未生成”，不伪装成成功
- 若步骤根本未进入，显示为“已跳过”
- 若关联 ID 为空数组，不显示空值占位

## 测试要求

前端至少覆盖以下场景：

- 评论已入库、已写记忆、已召回、已生成建议
- 评论已入库，但未产生记忆
- 评论已入库、已写记忆，但未命中召回
- 评论跳过召回
- 评论未生成建议
- 非评论事件不显示时间线
- 展开态能正确显示关联 ID 和说明

## 完成标准

满足以下条件即可视为完成：

- 评论卡片默认展示为横向时间线，而不是一排并列标签
- 展开后能看到纵向处理轨迹
- 用户能区分“成功、未命中、跳过”三类状态
- 所有时间线文案为中文
- 现有事件流功能和展开交互不回退
