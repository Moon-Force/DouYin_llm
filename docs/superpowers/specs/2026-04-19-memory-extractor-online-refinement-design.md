# 观众记忆在线提纯与即时生效设计

日期：2026-04-19

## 目标

把当前观众记忆抽取链路从“评论先生成建议、再抽取记忆”改成“评论先抽取并提纯候选记忆、再生成建议”，让同一条评论里刚暴露的新信息可以立刻参与当前建议生成。

本次设计同时解决三件事：

- 在已有“抽取前预筛”基础上，补上“抽取后提纯”这层门控
- 让新抽出的高质量记忆可以即时影响当前评论的建议
- 降低规则兜底权重，避免 LLM 明确拒绝的内容被规则误捞回长期记忆池

## 背景

当前主链路已经具备：

- 评论预筛，能挡掉明显寒暄、刷屏和纯交易问句
- LLM 记忆抽取协议，支持 `memory_type / polarity / temporal_scope`
- Ollama `/v1/chat/completions` 接入
- 离线抽取评测和边界测试集

但仍存在三个关键问题：

1. 当前评论不够“即时聪明”
   - 当前 `process_event` 先生成建议，再抽记忆
   - 结果是评论里刚暴露的新信息只能服务下一条评论，不能服务当下回复

2. 抽取结果缺少提纯层
   - 即使模型判断对了，也可能返回过长、过口语、过贴近原句的 `memory_text`
   - 直接落库会增加长期记忆污染，影响召回质量

3. 规则兜底过强
   - 当前只要 LLM 空结果或异常，就会回退规则抽取
   - 这会把本来应该被 LLM 判掉的内容重新捞回来

## 非目标

本次不做以下内容：

- 不引入礼物、进场、复购等新的记忆事件源
- 不实现同义记忆合并或长期记忆更新策略
- 不实现长期/短期双池存储
- 不修改向量召回排序算法
- 不在本轮做批量运营工作流或 UI 改造

## 方案比较

### 方案一：同步双阶段主链路

流程：

1. 评论入库
2. 记忆预筛
3. LLM 抽取并同时给出提纯结果
4. 严格门控
5. 合格候选记忆以“临时记忆”身份注入当前建议上下文
6. 生成建议
7. 正式持久化长期记忆并同步向量索引

优点：

- 当前评论立刻变聪明
- 长期记忆落库前还有一层提纯和门控
- 行为最符合直播互动目标

缺点：

- 单条评论同步路径更长，延迟上升

### 方案二：同步轻抽取，异步落库

优点：

- 当前建议也能吃到新信息
- 持久化可以放到后台，主链路更短

缺点：

- 当前建议上下文和最终落库结果可能短暂不一致
- 链路复杂度更高，状态诊断更难

### 方案三：仅做后处理提纯和规则兜底降权

优点：

- 风险最小
- 不会显著拉长主链路

缺点：

- 不能解决“当前评论立刻变聪明”

### 推荐方案

采用方案一：同步双阶段主链路。

原因：

- 用户已经明确优先“当前评论立刻变聪明”
- 本次改动的核心价值就在于把新信息即时反馈给当前建议
- 只要门控足够严，主链路更同步并不会直接导致长期记忆池变脏

## 总体设计

本次把抽取链路拆成三层：

1. 抽取前预筛层
2. LLM 抽取与提纯层
3. 持久化门控层

其中：

- 预筛层负责尽早挡掉明显垃圾流量，节省延迟和 token
- LLM 层负责理解原句并输出“原始抽取 + 提纯结果”
- 持久化门控层负责决定：
  - 哪些候选可以即时参与当前建议
  - 哪些候选可以正式进入长期记忆

## 一、双字段抽取协议

### 1. 协议目标

当前只有单个 `memory_text` 字段，不足以区分：

- 模型是否理解对了原句
- 模型是否把理解结果提纯得过头或不够

因此协议升级为双字段：

- `memory_text_raw`
- `memory_text_canonical`

### 2. 目标返回结构

LLM 返回的 JSON 字段改为：

```json
{
  "should_extract": true,
  "memory_text_raw": "我其实平时很少吃外卖，基本都是自己做饭",
  "memory_text_canonical": "平时很少吃外卖，基本自己做饭",
  "memory_type": "fact",
  "polarity": "neutral",
  "temporal_scope": "long_term",
  "reason": "稳定生活方式事实，对后续互动有复用价值"
}
```

### 3. 字段约束

- `memory_text_raw`
  - 当 `should_extract=true` 时必须为非空字符串
  - 允许更贴近原句表达
- `memory_text_canonical`
  - 当 `should_extract=true` 时必须为非空字符串
  - 必须是最小可复用表达
  - 用于当前建议、长期落库和后续召回
- `memory_text_raw`
  - 只用于日志、评测、排错和人工纠偏参考

### 4. 为什么保留双字段

双字段能清楚区分两类错误：

- 抽取错误：模型理解错了原句
- 提纯错误：模型理解对了，但提纯过头或保留了太多无关外壳

这对后续评测、运营排错和 prompt 调整都更友好。

## 二、LLM 提纯设计

### 1. 设计原则

提纯器不再用规则后处理硬裁剪，而是由同一次 LLM 调用直接输出 `memory_text_canonical`。

原因：

- 纯规则很难完整覆盖口语、省略、半句、反问、主语残缺等真实评论形态
- 同一次 LLM 输出比“先抽一次，再额外打一枪提纯”延迟更低

### 2. 提纯目标

`memory_text_canonical` 必须满足：

- 保留关键主语和限制条件
- 去掉无关口头语和互动壳子
- 尽可能短，但不能丢失复用所需信息
- 不保留问句壳子、反问壳子、情绪壳子

### 3. 典型案例

- `我其实平时吧都不太能吃辣`
  - raw：`我其实平时吧都不太能吃辣`
  - canonical：`不太能吃辣`

- `我租房住在公司附近，这样通勤方便点`
  - raw：`我租房住在公司附近，这样通勤方便点`
  - canonical：`租房住在公司附近`

- `这周我可能都在上海出差`
  - 应返回 `should_extract=false`
  - 不进入长期记忆

### 4. LLM 提纯失败处理

如果 LLM 返回：

- 缺字段
- `memory_text_canonical` 为空
- `memory_text_canonical` 明显仍是问句或纯互动句

则候选直接判为不合格，不再用规则提纯补救。

## 三、同步双阶段主链路

### 1. 当前顺序

当前 `process_event` 的顺序是：

1. 入库事件
2. 生成建议
3. 抽取记忆
4. 落库记忆

### 2. 新顺序

改成：

1. 入库事件
2. 预筛
3. LLM 抽取并提纯
4. 生成即时候选记忆
5. 把合格候选临时注入当前建议上下文
6. 生成建议
7. 持久化长期记忆
8. 同步向量索引
9. 发布事件和状态

### 3. 临时记忆与长期记忆

本次引入两个概念：

- `ephemeral memory`
  - 只用于当前评论建议生成
  - 先不落库
- `persisted memory`
  - 通过全部校验后再正式写入长期记忆表

### 4. 设计价值

- 当前评论立即吃到新信息
- 即使最后不落库，也可以让当前建议更贴身
- 长期池仍然由严格门控保护

## 四、规则兜底降权

### 1. 当前问题

当前只要 LLM 返回空结果，就会走规则兜底。

这会导致：

- LLM 明确判掉的内容，仍有机会被规则误入库
- 规则兜底实际变成第二主通路，而不是故障保险丝

### 2. 新的兜底原则

规则兜底只在“LLM 调用异常”时触发，不在“LLM 正常决策但不抽取”时触发。

具体约束：

- `should_extract=false`
  - 不走规则兜底
- LLM 成功返回但门控失败
  - 不走规则兜底
- 只有以下情况才允许规则兜底：
  - 请求超时
  - HTTP 异常
  - 返回非法 JSON
  - 返回结构缺失

### 3. 高置信模板限制

即使进入规则兜底，也不允许所有旧规则继续生效，而是只放行高置信模板：

- 主语明确
- 非问句
- 非短期状态
- 非情绪句
- 非互动句
- 稳定事实或稳定偏好一眼成立

例如：

- `我在杭州上班`
- `我家里养了两只猫`
- `我一直都只用安卓手机`

### 4. 结果

规则兜底从“第二主通路”降为“故障保险丝”。

## 五、当前建议上下文注入

### 1. Agent 上下文扩展

建议生成上下文除了已有的召回记忆，还允许附带当前评论刚抽出的临时记忆。

逻辑上：

- 旧记忆来自向量召回
- 新记忆来自当前事件同步抽取

### 2. 注入原则

只有满足以下条件的新记忆才允许即时注入：

- `should_extract=true`
- `temporal_scope=long_term`
- `memory_text_canonical` 非空
- 类型合法
- 没有明显问句/互动句污染

### 3. 排序原则

如果当前评论产出了新的 `ephemeral memory`：

- 它在当前建议里优先于旧召回记忆
- 但不直接覆盖旧记忆，只是作为上下文顶部补充

## 六、状态可观测性

为了方便排查新链路，评论处理状态需要新增字段：

- `memory_prefiltered`
- `memory_llm_attempted`
- `memory_refined`
- `memory_used_for_current_suggestion`
- `memory_persisted`

这样可以区分：

- 是否被预筛挡掉
- 是否真的调用了 LLM
- 是否拿到了有效提纯结果
- 当前建议是否即时吃到了这条新记忆
- 最终是否进入长期记忆

## 七、测试设计

### 1. 单测

需要新增或修改以下测试：

- 双字段协议校验
- `memory_text_raw / memory_text_canonical` 同时存在时的标准化
- canonical 为空时拒绝候选
- LLM 明确 `should_extract=false` 时不走规则兜底
- 只有 LLM 异常时才允许规则兜底
- 高置信模板规则兜底边界

### 2. 流程测试

需要覆盖：

- 当前评论新记忆是否参与本条建议
- 当前评论候选记忆不合格时，是否不会污染长期记忆
- 当前评论建议生成成功但记忆持久化失败时，事件是否仍正常发布

### 3. 评测扩展

离线评测需额外支持：

- `memory_text_raw` 与 `memory_text_canonical` 的观察输出
- 针对 `canonical` 的质量对比
- 统计“整句入库率”或“提纯失败率”

## 八、验收标准

完成后应满足：

1. 同一条评论里新抽出的高质量记忆能立即影响当前建议
2. 长期记忆落库只使用 `memory_text_canonical`
3. LLM 明确不抽时，规则不会再把内容捞回来
4. 只有 LLM 异常时，规则兜底才会介入
5. 抽取链路有可观测状态，可定位是预筛、LLM、提纯还是持久化出了问题
