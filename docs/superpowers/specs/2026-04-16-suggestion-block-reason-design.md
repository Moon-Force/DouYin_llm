# 未生成建议原因可见性设计

## 目标

让前端能够直接显示“这条评论为什么没有生成建议”，而不是让用户从是否存在建议、模型状态或日志里自行推断。

本次设计解决两个问题：

- 主视图里只能看出“生成了没有”，看不出“为什么没生成”
- 评论处理时间线已经能展示处理步骤，但还缺少“建议阶段”的明确结论和可读原因

## 适用范围

包含：

- 后端为每条评论的 `processing_status` 增加建议阶段的结构化状态字段
- 前端时间线主视图显示一个短结论
- 前端展开视图显示更完整的建议处理结果说明
- 为新增字段和展示文案补充测试

不包含：

- 新增独立的“失败评论列表”或诊断面板
- 让前端根据零散字段自行推导未生成原因
- 重做整个建议生成流程或引入新的模型策略

## 设计原则

- 原因由后端生成，前端负责展示，不做复杂推断
- 区分“跳过生成”和“生成失败”，避免语义混淆
- 主视图只给一个短结论，避免把评论列表变成诊断页
- 展开视图补充过程说明，让用户能看懂这一条评论经历了什么
- 先覆盖最常见原因码，保留后续扩展空间

## 方案比较

### 方案一：前端基于现有字段做推断

做法：

- 继续使用 `suggestion_generated`
- 前端结合模型状态、语义健康、是否命中规则来猜测原因

优点：

- 后端改动最小

缺点：

- 推断规则容易分散在多个组件里
- “跳过”和“失败”容易被猜错
- 后续新增原因时，前后端容易不同步

### 方案二：推荐方案，后端直接输出结构化建议状态

做法：

- 在 `processing_status` 中新增建议阶段字段
- 后端在评论处理结束时写入明确状态和原因
- 前端只负责映射成中文文案和时间线节点

优点：

- 责任边界清晰
- 前端实现简单且稳定
- 后续新增原因码时扩展成本低

缺点：

- 需要补齐后端多处分支的状态写入

### 方案三：只显示一段自由文本原因

做法：

- 后端只返回一个字符串，例如“语义后端不可用”

优点：

- 实现最快

缺点：

- 前端无法稳定区分状态类型
- 文案难以统一，也不利于测试

### 推荐方案

采用方案二：**后端输出结构化建议状态，前端做轻展示映射**。

## 数据结构设计

### 新增字段

在 `CommentProcessingStatus` 中新增：

- `suggestion_status`
- `suggestion_block_reason`
- `suggestion_block_detail`

建议含义如下：

- `suggestion_status`
  - `generated`：成功生成建议
  - `skipped`：本条评论被明确跳过，不进入生成结果
  - `failed`：尝试生成了，但最终失败
- `suggestion_block_reason`
  - 结构化原因码，供前端映射文案
  - 成功生成时可为空字符串
- `suggestion_block_detail`
  - 给用户看的简短补充说明
  - 允许直接显示在展开视图中

### 初始原因码范围

首批只覆盖以下几类：

- `semantic_backend_unavailable`
- `llm_failed`
- `rule_skipped`
- `no_generation_needed`

原因码语义边界：

- `semantic_backend_unavailable`
  - 语义召回依赖不可用，导致当前流程不能按预期完成建议生成
- `llm_failed`
  - 已尝试调用模型，但调用失败、返回异常或结果不可用
- `rule_skipped`
  - 由明确规则直接跳过，例如命中预设不生成条件
- `no_generation_needed`
  - 流程判断当前评论无需额外给主播建议，但不属于异常，也不是硬规则拦截

约束：

- `generated` 时，`suggestion_generated=true`，并带 `suggestion_id`
- `skipped` 或 `failed` 时，`suggestion_generated=false`
- `failed` 必须尽量带 `suggestion_block_reason`
- `skipped` 必须带明确原因码，不能只留空

## 后端设计

### 状态写入策略

后端不让前端猜，而是在评论处理链路结束时写入完整建议阶段结论：

- 成功生成建议：
  - `suggestion_status=generated`
  - `suggestion_generated=true`
  - `suggestion_block_reason=""`
  - `suggestion_block_detail=""`
- 因规则直接跳过：
  - `suggestion_status=skipped`
  - `suggestion_generated=false`
  - 填写对应的 `suggestion_block_reason`
  - `suggestion_block_detail` 使用可直接展示的中文短句
- 尝试生成但失败：
  - `suggestion_status=failed`
  - `suggestion_generated=false`
  - 填写失败原因码和一条简短说明

### 责任边界

建议生成服务负责产出“生成结果元数据”，评论处理入口负责把这部分元数据写回 `processing_status`。

这样做的原因：

- `LivePromptAgent` 最清楚是“跳过”“失败”还是“成功”
- 事件处理入口最适合统一落到 `processing_status`

### 细化约束

- 不把底层异常堆栈直接暴露给前端
- `suggestion_block_detail` 要面向业务可读，而不是技术日志
- 如果未来原因码增多，前端对未知原因码要有兜底文案

## 前端设计

### 主视图

每条评论在现有时间线摘要区域中增加一个“建议结果”结论：

- 已生成建议
- 已跳过生成
- 生成失败

如果未生成建议，则在结论旁或结论下方展示一条短原因，例如：

- 因语义后端不可用，未生成建议
- 本条评论被规则跳过
- 模型调用失败，未生成建议

主视图目标是让用户一眼知道结果，不要求一次性读完全过程。

### 展开视图

在评论的展开区域中，为“建议阶段”提供更完整说明：

- 展示阶段状态
- 展示中文原因标题
- 展示 `suggestion_block_detail`

如果成功生成建议，则保持现有成功态；如果未生成，则把“为什么没有”讲清楚。

### 文案映射

前端基于 `suggestion_status + suggestion_block_reason` 做中文映射。

映射原则：

- 主视图使用短句
- 展开视图可以稍长，但仍然保持运营可读
- 遇到未知原因码时，用统一兜底文案，例如“本条评论未生成建议，原因未标注”

## 时间线表现设计

评论处理时间线继续保留“接收、落库、记忆提取、记忆召回、建议生成”等阶段，但建议阶段改为更像结论节点：

- 成功：显示“已生成建议”
- 跳过：显示“已跳过生成”
- 失败：显示“生成失败”

展开后补充说明：

- 若是跳过，强调这是策略结果，不是系统坏掉
- 若是失败，强调这是尝试后失败，并给出简短原因

这样可以让用户把“评论处理轨迹”和“建议结果结论”一起看懂，而不是只看到一排标签。

## 测试设计

至少覆盖以下内容：

### 后端测试

- 成功生成建议时，新增字段写入正确
- 规则跳过时，状态为 `skipped` 且原因码正确
- 模型失败时，状态为 `failed` 且原因码正确

### 前端测试

- presenter 能把 `generated / skipped / failed` 映射成正确中文状态
- presenter 能把原因码映射成主视图短文案和展开说明
- 未知原因码时有兜底文案
- 时间线组件在未生成建议时能显示原因说明

## 完成标准

满足以下条件即可视为完成：

- 后端 `processing_status` 提供结构化建议阶段状态
- 前端主视图能直接看出“为什么没生成建议”
- 前端展开视图能展示更完整的建议阶段说明
- 跳过与失败在前端表现上有明确区别
- 新增状态字段和文案映射有测试覆盖
- 本功能单独提交，不混入其他无关改动
