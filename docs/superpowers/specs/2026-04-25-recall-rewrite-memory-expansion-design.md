# 召回重写与记忆扩写设计

## 目标

提升观众长期记忆的语义召回质量：存储给人看的记忆保持简短准确，同时为向量检索维护一份更适合 embedding 的召回文本，并在查询侧补充检索意图重写。

## 设计

- `memory_text` 继续作为 UI、提示词和人工编辑使用的 canonical 记忆，不做扩写污染。
- 新增 `memory_recall_text` 作为专用 embedding 文本；为空时回退 `memory_text`，兼容历史数据。
- 新记忆创建、证据合并、记忆升级时优先生成 LLM 召回扩写；LLM 不可用或返回无效内容时使用规则扩写 fallback。
- 观众发言进入向量召回前生成 `query_recall_text`，召回查询使用原句和重写文本的组合，避免只靠表面措辞匹配。
- Chroma document 和 embedding 使用 `memory_recall_text`，返回给业务和前端的仍是 `memory_text`。

## 非目标

- 不修改已有 `memory_text` 的展示语义。
- 不让扩写内容直接进入主播回复提示中的记忆列表，避免幻觉扩写被展示给用户。
- 不引入额外向量库或外部依赖。

## 风险与保护

- LLM 扩写可能引入过度推断，因此扩写文本只能用于召回，不替代事实记忆。
- LLM 调用失败不能影响主流程，必须 fallback 到规则扩写。
- 旧 Chroma 索引需要因 document 变化自动重建，启动采样校验会比较新的召回文本。
