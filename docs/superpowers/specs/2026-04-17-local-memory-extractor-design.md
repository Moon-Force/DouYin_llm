# 本地记忆抽取器设计

日期：2026-04-17

## 目标

将观众记忆抽取从“纯规则关键词抽取”升级为“项目内嵌本地大模型主抽取 + 规则兜底”的架构，满足以下约束：

- 不依赖云端 LLM
- 不要求用户额外启动 `Ollama` 或其他独立模型服务
- 模型在项目进程内运行时加载
- 面向 Windows + CPU 的广泛使用场景
- 优先复用本地模型文件，缺失时自动下载
- 第一版只用于观众记忆抽取，不改造提词生成链路

## 背景

当前项目已经具备完整的“评论入库 -> 记忆保存 -> 语义召回 -> 提词生成”主链路，但观众记忆抽取仍主要依赖长度、关键词和少量类型规则。这样虽然可运行，但存在几个核心问题：

- 项目已经集成了 LLM 和 embedding，记忆抽取却没有利用 LLM 的语义理解能力
- 抽取结果更接近“整句评论入库”，而不是“提炼后的关键记忆”
- 短期状态、问句、互动句和低价值内容可能进入长期记忆池
- 对负向偏好、不确定表达、隐含偏好和复合句的处理能力较弱

对于本项目的目标来说，记忆抽取的第一优先级不是“尽量多记”，而是“尽量少错记”。因此第一版改造重点是提升长期记忆的质量，而不是扩大入库规模。

## 非目标

本次设计不包含以下内容：

- 不改造直播提词生成所使用的 LLM 调用方式
- 不要求将所有现有云端 LLM 路径切换为本地模型
- 不引入 `Ollama`、`vLLM`、`LM Studio` 等外部模型服务依赖
- 不在第一版中处理礼物事件、复购行为或其他非评论事件的记忆抽取
- 不在第一版中实现语义合并、记忆去重升级、时间衰减落库或批量回填旧数据
- 不在第一版中修改 `viewer_memories` 表结构

## 方案比较

### 方案一：继续使用云端 OpenAI-compatible LLM 做记忆抽取

优点：

- 改动最小，可快速复用现有 HTTP 调用方式
- 推理质量上限较高

缺点：

- 违背“项目内运行、无需云端”的明确目标
- 不能满足宽泛 CPU 场景下的离线使用要求

### 方案二：在项目内使用 `transformers + torch` 直接加载本地模型

优点：

- 纯 Python 方案，理论上兼容主流 Hugging Face 模型
- 后续扩展空间大

缺点：

- Windows + CPU 场景下安装和运行负担较重
- 启动时间、内存压力和推理延迟都不理想
- 不适合作为“广泛可用”的默认方案

### 方案三：在项目内使用 `llama-cpp-python + GGUF` 加载本地量化模型

优点：

- 不依赖外部推理服务
- 更适合 Windows + CPU 场景
- 支持项目内直接加载，符合“没有就下载，有就复用”的模型管理方式
- 对短 JSON 输出任务较友好，适合本项目的记忆抽取需求

缺点：

- 需要明确约束模型格式为 GGUF
- 需要在项目中增加本地模型下载、缓存和加载管理

### 推荐方案

采用方案三：`llama-cpp-python + GGUF` 内嵌本地模型，作为记忆抽取主引擎，并保留当前规则抽取作为兜底。

原因：

- 与“Windows + CPU 优先、项目内运行、不用开服务”的要求最匹配
- 改造范围清晰，能在不打断现有提词链路的前提下独立演进
- 本地模型不可用时仍可回退到当前规则，降低上线风险

## 总体架构

保留 `ViewerMemoryExtractor.extract(event)` 作为唯一入口，但将内部实现拆分为三层：

1. `LocalMemoryExtractionModel`
   负责模型路径解析、自动下载、懒加载和单次推理调用。

2. `LLMBackedViewerMemoryExtractor`
   负责 prompt 拼装、JSON 结果解析、字段校验、后处理和记忆候选构建。

3. `RuleFallbackMemoryExtractor`
   复用当前规则抽取逻辑，作为本地模型不可用、推理失败、超时或结果非法时的兜底路径。

对外流程保持不变：

1. `process_event()` 调用 `memory_extractor.extract(event)`
2. `extract(event)` 内部优先尝试本地 LLM 抽取
3. 成功则返回结构化记忆候选
4. 失败则回退规则抽取
5. `process_event()` 继续调用 `save_viewer_memory(...)`

这样可以将改动局限在抽取层，不破坏事件处理、入库、召回和提词生成的主链路。

## 模型文件与运行时管理

### 目录结构

项目内固定模型目录：

```text
data/models/memory_extractor/
```

默认模型文件名采用单文件 GGUF，例如：

```text
qwen2.5-3b-instruct-q4_k_m.gguf
```

### 配置项

新增以下配置项：

- `MEMORY_EXTRACTOR_ENABLED=true`
- `MEMORY_EXTRACTOR_MODE=local`
- `MEMORY_EXTRACTOR_MODEL_PATH=`
- `MEMORY_EXTRACTOR_MODEL_URL=`
- `MEMORY_EXTRACTOR_MODEL_FILENAME=memory-extractor.gguf`
- `MEMORY_EXTRACTOR_CONTEXT_SIZE=4096`
- `MEMORY_EXTRACTOR_MAX_TOKENS=512`
- `MEMORY_EXTRACTOR_TIMEOUT_SECONDS=30`
- `MEMORY_EXTRACTOR_THREADS=`

规则：

- `MODEL_PATH` 优先级最高
- 未显式配置路径时，使用默认目录 + 默认文件名
- 本地文件不存在且配置了 `MODEL_URL` 时，自动下载到默认目录
- 下载成功后复用本地文件，后续不重复下载

### 加载策略

采用懒加载：

- 应用启动时不强制加载模型
- 第一次真正触发记忆抽取时，才检查文件并加载模型
- 模型实例在进程内缓存复用

这样可以避免后端启动时因为模型加载拖慢初始化，也更适合非记忆抽取测试场景。

### 失败处理

当出现以下情况时，不中断评论处理主链路：

- 模型文件不存在且下载失败
- 模型加载失败
- 推理超时
- 返回结果不是合法 JSON
- 返回 JSON 结构不合法

上述情况统一记录日志，并切换到规则兜底抽取。

## LLM 抽取输出协议

本地模型只允许输出一个小而硬的 JSON 对象，不允许输出解释、markdown 或多段文本。

目标结构：

```json
{
  "should_extract": true,
  "memory_text": "喜欢豚骨拉面",
  "memory_type": "preference",
  "polarity": "positive",
  "temporal_scope": "long_term",
  "certainty": "high",
  "reason": "稳定偏好，适合主播后续接话"
}
```

字段定义：

- `should_extract`
  是否值得沉淀为长期记忆
- `memory_text`
  提炼后的关键记忆，不能是整句评论原样照抄
- `memory_type`
  第一版仅允许：`preference | fact | context`
- `polarity`
  `positive | negative | neutral`
- `temporal_scope`
  `long_term | short_term`
- `certainty`
  `high | medium | low`
- `reason`
  仅用于调试和日志，不入库

## 第一版入库门槛

为了优先提升长期记忆质量，第一版的入库条件定为：

- `should_extract` 必须为 `true`
- `memory_text` 不能为空，且长度必须在合理范围内
- `memory_type` 必须在允许集合内
- `temporal_scope` 必须为 `long_term`
- `certainty` 不能为 `low`

这些规则意味着：

- 问句、互动句、试探句更容易在抽取阶段被拦截
- 短期计划和短期状态不会进入长期记忆表
- 第一版优先保留长期稳定、对主播有复用价值的信息

## 置信度策略

第一版不直接信任模型自报的数值置信度，而是让模型返回离散的 `certainty`，再由代码统一映射：

- `high` -> `0.82`
- `medium` -> `0.68`
- `low` -> 不入库

随后再做少量后处理修正：

- `memory_type` 为 `preference` 或 `context` 时可小幅加成
- `memory_text` 过长时降分
- 看起来像原句整抄时降分
- 最终分数限制在 `0~1`

这样做的目的，是避免把“模型主观感觉的分数”直接当作稳定业务信号。

## 规则兜底策略

规则兜底保留现有 `ViewerMemoryExtractor` 的规则版本，但触发条件明确限制为：

- 本地模型不可用
- 模型下载失败
- 模型加载失败
- 模型推理超时
- 返回非 JSON
- 返回 JSON 缺字段或字段非法

规则兜底的角色不是追求高质量，而是保证本地模型不可用时，主链路仍然可运行。

## 提示词设计原则

本地模型用于记忆抽取时，prompt 要明确约束以下原则：

- 你的任务是“抽取对主播后续互动有用的长期记忆”，不是总结整条评论
- 只抽长期稳定信息，不抽短期状态或临时计划
- 问句、反问、互动试探、让主播猜的问题通常不应进入长期记忆
- 对“不喜欢”“不要推荐”“吃不了”“过敏”等表达，要识别为负向偏好或负向事实
- `memory_text` 要尽量短、可复用、主播能直接理解

## 对现有链路的影响

### 不变的部分

- `process_event()` 的主流程保持不变
- `save_viewer_memory(...)` 的调用方式保持不变
- `viewer_memories` 表结构第一版保持不变
- 语义召回和提词生成链路不做本轮改造

### 会变化的部分

- 自动抽取的候选记忆来源从“纯规则”变为“本地 LLM 优先”
- `memory_text` 的风格从“原评论整句”转为“提炼后的关键记忆”
- 长期记忆的入库数量可能减少，但质量应提升

## 测试设计

至少补齐以下测试：

1. 本地模型成功返回合法 JSON 时，`extract(event)` 生成候选记忆
2. 返回 `short_term` 时，不返回长期记忆候选
3. 返回负向偏好时，能保留负向表达，不误写成正向偏好
4. 返回非法 JSON 时，自动回退规则抽取
5. 模型文件不存在但配置了下载地址时，会自动下载
6. 本地模型文件已存在时，不重复下载
7. 模型推理超时或加载失败时，不影响整条评论处理链路
8. `process_event()` 仍能正常保存由本地模型抽出的候选记忆

## 风险与缓解

### 风险一：CPU 推理延迟过高

缓解：

- 第一版只用于记忆抽取，不影响提词生成路径
- 使用量化 GGUF 模型
- 限制 `max_tokens`
- 仅在评论事件触发抽取时调用

### 风险二：模型输出不稳定

缓解：

- 严格 JSON 协议
- 解析失败即回退规则
- 第一版只允许有限字段和有限类型

### 风险三：自动下载失败影响用户体验

缓解：

- 优先使用本地文件
- 缺失时才自动下载
- 下载失败仅影响 LLM 抽取，不阻塞整条评论处理

## 实施顺序建议

1. 扩展配置项和模型目录解析逻辑
2. 新增本地模型下载与懒加载组件
3. 新增 LLM 驱动的记忆抽取器
4. 将现有规则抽取收敛为明确的兜底实现
5. 在 `app.py` 中装配新的抽取器
6. 补齐抽取层与流程层测试
7. 更新 README 和 `.env.example`

## 完成标准

满足以下条件即可视为本轮完成：

- 项目在 Windows + CPU 场景下可在进程内直接加载本地记忆抽取模型
- 本地模型文件存在时直接复用，缺失时可自动下载
- 记忆抽取优先使用本地模型，不依赖云端，也不需要用户手动开外部服务
- 本地模型不可用时，规则兜底仍能保证主链路可运行
- 第一版只影响观众记忆抽取，不影响提词生成路径
- 有覆盖关键路径的自动化测试
